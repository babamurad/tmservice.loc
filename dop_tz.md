# Дополнительное ТЗ: исправление упущений основного плана

> Этот документ дополняет основной «План разработки сервиса». Он не заменяет его, а закрывает дыры в безопасности, архитектуре и функционале. Каждый раздел ссылается на шаги исходного плана, которые он меняет или дополняет.

**Порядок внедрения по приоритету:**
1. **Блок A (критичное, безопасность)** — обязательно до публичного запуска.
2. **Блок B (архитектурные исправления)** — до того, как написан зависимый код, иначе придётся переделывать.
3. **Блок C (пропущенный функционал)** — в рамках MVP.
4. **Блок D (продуктовое)** — настоятельно рекомендуется в MVP, можно сразу после.

## Блок A. Критичное — безопасность (до запуска)

### Этап A1. Защита назначения роли при регистрации

**Проблема.** В исходном Шаге 2.2 `role` приходит из тела запроса и валидируется как обычное поле. Любой клиент может зарегистрироваться как `admin`, подставив роль в JSON — это полный доступ к будущей админ-панели за один POST.

- [ ] Шаг A1.1: Убрать `role` из принимаемых при регистрации полей. Сервер сам решает роль, а не клиент.
- [ ] Шаг A1.2: Разрешить при публичной регистрации только `client` и `master`.
  - Вариант 1 (рекомендуется): два отдельных эндпоинта — `POST /api/register/client` и `POST /api/register/master`, роль зашита в контроллере.
  - Вариант 2: один эндпоинт с валидацией `'role' => 'required|in:client,master'`. `admin` в `in:` НЕ входит никогда.
- [ ] Шаг A1.3: Роль `admin` создаётся **только** через сидер или artisan-команду (`php artisan make:admin {phone}`), но не через HTTP.
- [ ] Шаг A1.4: Вынести валидацию регистрации в отдельный `FormRequest` (`RegisterRequest`), чтобы правило роли не «потерялось» при рефакторинге контроллера.

**Критерии приёмки:**
- Запрос на регистрацию с `"role": "admin"` возвращает `422` или роль игнорируется и ставится `client`/`master`.
- Создать админа через публичный API невозможно ни при каком теле запроса.

### Этап A2. Верификация телефона (OTP)

**Проблема.** Телефон — основной идентификатор и главное действие приложения (звонок мастеру), но при регистрации он ничем не подтверждается. Без OTP каталог забьётся визитками на чужих и несуществующих номерах.

> ⚠️ **Открытый вопрос — SMS-шлюз для Туркменистана.** Доставка SMS на номера `+993` через международные шлюзы (Twilio и т.п.) может быть нестабильной или недоступной — телеком в стране ограничен. Это потенциальное узкое место всего продукта, а не просто фича. **Прежде чем писать OTP-логику, нужно провести отдельный спайк:** проверить реальную доставку через 1–2 кандидата-шлюза на живой `+993`-номер. Запасные варианты — в Шаге A2.5.

- [ ] Шаг A2.1: Таблица `phone_verifications`
  - Поля: `id`, `phone` (string, index), `code` (string, 4–6 цифр), `expires_at` (timestamp), `attempts` (unsigned int, default 0), `timestamps`.
- [ ] Шаг A2.2: Эндпоинт `POST /api/auth/send-otp`
  - Принимает `phone`. Генерирует код, пишет в `phone_verifications`, отправляет SMS. Код живёт 5 минут.
- [ ] Шаг A2.3: Эндпоинт `POST /api/auth/verify-otp`
  - Принимает `phone` + `code`. Сверяет код и срок. Лимит `attempts` — например, 5 попыток, потом код инвалидируется.
- [ ] Шаг A2.4: Изменить флоу регистрации (Шаг 2.2): пользователь создаётся, но активным/видимым в каталоге становится только после успешной верификации. Добавить `users.phone_verified_at` (nullable timestamp).
- [ ] Шаг A2.5: Слой отправки SMS сделать через интерфейс `SmsGateway` с несколькими реализациями, чтобы шлюз можно было заменить, не трогая контроллеры:
  - `LogSmsGateway` — для local/staging пишет код в лог вместо реальной отправки.
  - Реальный провайдер — определяется по итогам спайка из «открытого вопроса».
  - **Запасной план**, если SMS-доставка к `+993` окажется нерабочей: ручная модерация мастеров админом, либо верификация звонком (flash-call), либо подтверждение через мессенджер. Решение принять до старта разработки этапа.
- [ ] Шаг A2.6: Антиспам на отправку OTP — не чаще 1 кода в 60 секунд на номер и не более N в час (см. Этап A3).

**Критерии приёмки:**
- Зарегистрироваться без подтверждённого кода и попасть в выдачу мастеров нельзя.
- На local-окружении код виден в логах, реальные SMS не уходят.
- Подбор кода ограничен по числу попыток и времени жизни.

### Этап A3. Rate limiting на публичные и auth-маршруты

**Проблема.** `GET /api/masters/{id}` отдаёт телефон публично — by design, но это значит, что спарсить базу всех номеров тривиально. Плюс `login` и `send-otp` без лимитов открыты для брутфорса и SMS-флуда (а SMS — это ещё и деньги).

- [ ] Шаг A3.1: Описать именованные лимитеры в `App\Providers` (или `bootstrap/app.php` для Laravel 11+):
  ```php
  RateLimiter::for('public-read', fn($r) => Limit::perMinute(60)->by($r->ip()));
  RateLimiter::for('auth',        fn($r) => Limit::perMinute(5)->by($r->ip()));
  RateLimiter::for('otp',         fn($r) => Limit::perMinutes(60, 5)->by($r->input('phone')));
  ```
- [ ] Шаг A3.2: Навесить `throttle:public-read` на `GET /api/masters` и `GET /api/masters/{id}`.
- [ ] Шаг A3.3: Навесить `throttle:auth` на `login` и `register`, `throttle:otp` — на `send-otp`/`verify-otp`.

**Критерии приёмки:**
- Превышение лимита возвращает `429` с заголовком `Retry-After`.
- Массовый перебор `masters/{id}` упирается в лимит.

## Блок B. Архитектурные исправления (до зависимого кода)

### Этап B1. QR через Universal Links / App Links вместо кастомной схемы

**Проблема.** В исходном Шаге 4.2 QR кодирует `appscheme://master/{id}`. Кастомная схема срабатывает, только если приложение уже установлено. Клиент сканирует визитку мастера без установленного приложения — и ссылка молча не делает ничего. Для QR, смысл которого в привлечении новых клиентов, это провал сценария.

- [ ] Шаг B1.1: QR кодирует **https-ссылку** на домене сервиса: `https://{домен}/m/{id}`, а не кастомную схему. Изменить Шаг 4.2 основного плана соответственно.
- [ ] Шаг B1.2: Веб-страница `/m/{id}` на бэкенде (можно простой Blade — это единственное исключение из «только JSON»):
  - Если приложение установлено → ОС сама откроет его (по Universal Links / App Links).
  - Если нет → показывается карточка мастера + кнопки «Открыть в приложении / Скачать в App Store / Google Play».
- [ ] Шаг B1.3: Бэкенд отдаёт файлы ассоциации доменов:
  - iOS: `https://{домен}/.well-known/apple-app-site-association` (без расширения, `Content-Type: application/json`).
  - Android: `https://{домен}/.well-known/assetlinks.json`.
- [ ] Шаг B1.4: В Expo (`app.json`/`app.config.js`) настроить:
  - iOS — `associatedDomains: ["applinks:{домен}"]`.
  - Android — `intentFilters` с `autoVerify: true` на `https`-схему и хост домена.
- [ ] Шаг B1.5: В приложении обработать входящий линк через `expo-linking` и смаппить `/m/{id}` на экран `MasterDetailScreen`.

**Критерии приёмки:**
- Скан QR при установленном приложении → открывается `MasterDetailScreen` нужного мастера.
- Скан QR без приложения → открывается веб-страница с переходом в стор, а не «пустота».

### Этап B2. Offline-first: разделить серверный кэш и клиентский персист

**Проблема.** В исходном Шаге 3.1 серверное кэширование (Redis/File) названо способом сделать «offline-first». Это разные слои: серверный кэш снижает нагрузку на БД, но к офлайну в мобилке отношения не имеет. Offline-first — это персист данных на устройстве.

- [ ] Шаг B2.1: Серверный кэш справочников (`cities`, `categories`) оставить — он полезен, но как оптимизация нагрузки на БД, не как «offline». Добавить инвалидацию кэша при изменении справочников из админки.
- [ ] Шаг B2.2: Клиентский offline-first (это новое):
  - Хранить `cities`/`categories` на устройстве (`AsyncStorage`, либо persist-плагин TanStack Query / Redux Persist).
  - При старте показывать данные из локального кэша мгновенно, в фоне обновлять с сервера.
- [ ] Шаг B2.3: Версионирование справочников для дешёвой синхронизации: эндпоинт отдаёт `version`/`updated_at`, клиент перекачивает данные, только если версия изменилась (или используем `ETag` / `If-None-Match` → `304`).

**Критерии приёмки:**
- Открытие каталога без сети показывает ранее загруженные города/категории.
- Повторный запрос без изменений на сервере не тянет полный объём данных.

## Блок C. Пропущенный функционал (в рамках MVP)

### Этап C1. Экран поиска (`SearchTab`)

**Проблема.** `SearchTab` есть в навигации (Шаг 5.3), но экрана под него в исходном Этапе 6 нет — таб ведёт в никуда.

Принять одно из решений:

- [ ] Шаг C1.1 (вариант А — реализовать): `SearchScreen`
  - Поле текстового поиска + фильтры по городу/категории.
  - Запрос `GET /api/masters?q={строка}&city_id=&category_id=`.
  - На бэкенде расширить `MasterController@index`: поиск по `bio` и имени мастера (`LIKE`/полнотекстовый), фильтры комбинируются с уже описанными `city_id`/`category_id`.
- [ ] Шаг C1.2 (вариант Б — убрать): удалить `SearchTab` из `Bottom Tabs`, если поиск не входит в MVP. Не оставлять «мёртвый» таб.

**Критерии приёмки:**
- Таб `Search` либо открывает рабочий поиск, либо отсутствует.

### Этап C2. Админ-панель и API для роли `admin`

**Проблема.** Роль `admin` заведена в модели User, но под неё нет ни одного эндпоинта, ни интерфейса. Сейчас админ ничего не умеет, хотя справочниками (города/категории) и модерацией кто-то должен управлять.

- [ ] Шаг C2.1: Выбрать инструмент:
  - **Рекомендуется Filament** — готовая админка на Laravel, быстро закрывает CRUD без отдельного фронта.
  - Альтернатива — защищённые `admin`-маршруты API + middleware/Gate по `role === 'admin'`, если админку решат делать своим фронтом.
- [ ] Шаг C2.2: Функционал админки (минимум):
  - CRUD городов и категорий (с переключателем `is_active`) → при изменении сбрасывать серверный кэш из B2.1.
  - Список мастеров с модерацией: блокировка/разблокировка, просмотр жалоб.
  - Просмотр пользователей по роли.
- [ ] Шаг C2.3: Добавить мастерам статус модерации: `master_profiles.moderation_status` (enum: `pending`, `approved`, `rejected`, default `pending`). В публичную выдачу (`GET /api/masters`) попадают только `approved`. Это же закрывает запасной сценарий из A2.5 (ручная верификация).
- [ ] Шаг C2.4: Защитить доступ: middleware, пропускающий только `admin`. Любой не-admin → `403`.

**Критерии приёмки:**
- Админ может включать/выключать города, категории и одобрять/блокировать мастеров.
- Неодобренный мастер не виден в публичном каталоге.

### Этап C3. Управление портфолио (удаление/замена)

**Проблема.** В исходном Шаге 7.2 фото только загружается. Удалить или заменить картинку мастер не может — попросят почти сразу.

- [ ] Шаг C3.1: Эндпоинт `DELETE /api/profile/portfolio/{id}` (`auth:sanctum`).
  - Проверка владения: фото принадлежит профилю текущего мастера, иначе `403`.
  - Удалять и файл из `storage`, и запись в `portfolio_images`.
- [ ] Шаг C3.2: Ограничение на число фото (например, не более 10 на профиль) — проверять при загрузке в Шаге 3.3 / 7.2.
- [ ] Шаг C3.3: В приложении (`MyProfileScreen`) — кнопка удаления на каждой картинке галереи.

**Критерии приёмки:**
- Мастер удаляет своё фото; чужое удалить не может.
- Загрузка сверх лимита возвращает понятную ошибку.

## Блок D. Продуктовое — отзывы и рейтинги (рекомендуется в MVP)

**Проблема.** В сервисе «найди мастера» доверие — это весь продукт. Сейчас клиент выбирает ни на чём: имя + точка «свободен/занят». Но «зелёная точка» — это про занятость, а не про качество работы. Без рейтинга у клиента нет сигнала, криворукий мастер или нет.

> **Открытый вопрос — кто может оставлять отзыв.** Подтвердить факт оказанной услуги в этой модели технически нечем (звонок идёт через системную звонилку, сделок в приложении нет). Варианты: (а) отзыв от любого верифицированного клиента, один на пару клиент↔мастер; (б) отложить до появления заявок/заказов в приложении. Для MVP проще (а) + модерация. Решение зафиксировать до реализации.

- [ ] Шаг D1: Модель `Review`
  - Связи: `belongsTo(User as client)`, `belongsTo(MasterProfile)`.
  - Поля: `id`, `client_id` (fk), `master_profile_id` (fk), `rating` (unsigned tinyint, 1–5), `comment` (text, nullable), `moderation_status` (enum: `pending`/`approved`/`rejected`), `timestamps`.
  - Уникальный индекс `(client_id, master_profile_id)` — один отзыв на пару.
- [ ] Шаг D2: Денормализованные поля в `master_profiles` для быстрой выдачи: `avg_rating` (decimal, default 0), `reviews_count` (unsigned int, default 0). Пересчитывать при добавлении/одобрении/удалении отзыва (через observer или событие).
- [ ] Шаг D3: API
  - `POST /api/masters/{id}/reviews` (`auth:sanctum`, только `client`, только верифицированный) — создаёт отзыв со статусом `pending`.
  - `GET /api/masters/{id}/reviews` — список одобренных отзывов с пагинацией.
  - В `GET /api/masters` и `GET /api/masters/{id}` добавить `avg_rating` и `reviews_count`.
- [ ] Шаг D4: Сортировку в `GET /api/masters` дополнить: сначала `is_free = true`, затем по `avg_rating` убыванием.
- [ ] Шаг D5: Модерация отзывов в админке (Этап C2) — одобрение/скрытие, чтобы убрать накрутку и оскорбления.
- [ ] Шаг D6: В приложении — звёзды и количество отзывов в карточке (`MastersListScreen`) и блок отзывов на `MasterDetailScreen` + форма «оставить отзыв».

**Критерии приёмки:**
- Клиент оставляет отзыв с оценкой 1–5; повторно по тому же мастеру — нельзя.
- Рейтинг и число отзывов видны в списке и в карточке.
- Неодобренные отзывы публично не показываются.

## Открытые вопросы для решения до старта
1. **SMS-шлюз для `+993`** — провести спайк на доставку; выбрать запасной механизм верификации (A2).
2. **Инструмент админки** — Filament или собственный фронт на API (C2.1).
3. **Кто может оставлять отзыв** — любой верифицированный клиент или ждём заявки в приложении (Блок D).
